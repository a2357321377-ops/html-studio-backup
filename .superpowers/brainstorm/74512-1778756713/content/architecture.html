架构方案对比

三种不同的技术架构，各有权衡

A

Monolith 全栈 — Next.js App Router

Next.js 前后端一体，API Routes 处理文件上传/AI调用，React Server Components 渲染编辑器

B

前后端分离 — Vite + Express

Vite React 前端 + Express/Fastify 后端，独立开发部署，灵活度高

C

Monorepo 全栈 — Vite + Hono

Turborepo monorepo，Vite React 前端 + Hono 轻量后端，共享类型定义，现代工具链

方案 A：Next.js App Router

优势

  • 前后端一体，部署简单（Vercel 一键）
  • API Routes 天然支持文件上传/AI 代理
  • 社区生态成熟，组件库丰富
  • SSR 可优化首屏加载

劣势

  • App Router 学习曲线较陡
  • 对复杂客户端编辑器场景，SSR 价值有限
  • 自定义部署不如独立后端灵活
  • html-ppt 的静态资源集成需要额外处理

方案 B：Vite + Express

优势

  • 前后端完全解耦，独立迭代
  • Vite 开发体验极佳，HMR 快
  • Express 生态成熟，中间件丰富
  • 部署灵活，可分别扩展

劣势

  • 需要维护两套项目配置
  • 类型不共享，前后端需手动同步
  • CORS 配置等额外工作
  • 部署时需要两个服务

方案 C：Vite + Hono（推荐）

优势

  • Monorepo 共享类型，前后端类型安全
  • Hono 超轻量，启动快，Edge 友好
  • Vite 开发体验好，HMR 快
  • 可部署到 Vercel/Cloudflare/Node
  • html-ppt 静态资源可直接作为 public assets

劣势

  • Hono 生态不如 Express 成熟
  • Monorepo 初始配置稍复杂
  • 团队需熟悉 Turborepo 工作流

推荐：方案 C（Vite + Hono Monorepo）

理由：这个项目的核心是「编辑器 + 文件处理 + AI 调用」，前后端类型共享很重要（模板定义、动效配置等）。Hono 足够轻量，不会引入不必要的复杂度。Monorepo 让 html-ppt 的 36 个主题、31 个布局、47 个动效可以作为共享包被前后端同时使用。Vite 的快速 HMR 对编辑器开发体验至关重要。
